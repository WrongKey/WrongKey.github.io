DEVOPS 工具篇之Ansible基础篇

关于自动化配置工具

使用传统方式(shell/bat)管理多主机集群存在的问题

  • 难以复用和维护
  • 极易出错，差异性难以避免
  • 对人的依赖性大
  • 逐步复杂，可读性有限
  • 不可跨平台
  • 对管理、运行人员要求高

自动化配置管理带来的好处

  • 使用第三方工具，支持跨平台。
  • 专有的DSL，更高的可读性，可重用性高，可审查性高。
  • 高效的集群操作，易于管理，保证集群环境的一致性。

什么是Ansible

我们先来了解一下什么是Ansible，Ansible官网上是这样描述这个工具的：

Ansible是一个Python实现的基于SSH协议的自动化配置管理工具，其主要工作原理是在一组主机（inventory）上执行一系列预先定义好的操作（playbook）。

这个工具主要可以帮助我们完成以下几个功能：

  • Configuration Management
  • Application Deployment
  • Multi-Tier Orchestration

为什么选择Ansible

Ansible的特点

Simple

易理解，易编写，易维护且无需编写脚本和自定义代码，能够很快的学习和上手使用。

Efficient

基于OpenSSH, 受控机器无需安装Agent。

Secure

使用SSH传输数据，无Agent。

Ansible相关概念

Inventory

Ansible提供了inventory文件是用来指定在哪些受控主机执行自动化任务。

inventory文件使用的是ini文件格式。方括号[]中是组名,用于对主机进行分类,便于对不同系统进行个别的管理.

如果有主机的SSH端口不是标准的22端口,可在主机名之后加上端口号,用冒号分隔.

通过设置针对组或主机的参数,可以控制 ansible 与远程主机的交互方式。

Module

Ansible通过Module来对需要执行在不同主机上的任务进行一层抽象，可以一定程度屏蔽不同系统的差异。

常用模块：

  • command 模块 —— 运行指定命令
  • ping 模块 —— 测试主机是否连通
  • shell 模块 —— 运行shell脚本
  • apt/yum 模块 —— 包管理器来管理软件包

使用ping模块来测试主机是否可访问。

ad-hoc命令

假如我一时兴起，想去ping一下某一台服务器，我就直接敲ansible all -i inventory.ini -m ping，这条命令不会被保存起来, 这样的命令就叫做 ad-hoc 命令.

Ansible提供两种方式去完成任务,一是 ad-hoc 命令,一是写 Ansible playbook.前者可以解决一些简单的任务, 后者解决较复杂的任务.

Playbook

Playbooks 与 adhoc 相比,是一种完全不同的运用 ansible 的方式,是非常之强大的.

简单来说,playbooks 是一种简单的配置管理系统与多机器部署系统的基础.与现有的其他系统有不同之处,且非常适合于复杂应用的部署.

Play

一个Playbook由一系列的Plays构成，每一个Play又包含了一系列的Tasks,每一个Task通过对 ansible模块的调用来完成特定的任务。

Play通常由name, hosts（指定在哪些主机上执行）,arguments(remote_user, sudo), varibles以及tasks。同时你也可以指定某个task只在某种条件下执行。

我是这样理解的，Playbook就是一个剧本，里面写了很多剧（Play）,每一个剧里面包含了很多需要我们的英雄(Ansible)要去执行的任务（Tasks）。

Handlers也是一些task的列表,可以通过名字来引用,它们和一般的task并没有什么区别.Handlers是由通知者进行notify, 如果没有被notify,handlers不会执行.不管有多少个通知者进行了notify,等到play中的所有task执行完成之后,handlers也只会被执行一次，handlers会按照声明的顺序执行.

AWS之IAM

最近一段时间一直在学习AWS相关的东西，近两周把IAM（Identity and Access Management）的文档过了一遍, 现在总结一下学到的东西。

IAM定义:

通过 IAM 可以控制哪些人可以使用您的 AWS 资源（身份验证）以及他们可以使用的资源和采用的方式（授权）。

其实，这就和Linux的权限系统很相似，只是将对文件系统的操作换成了对AWS上的资源的操作。我们在注册成为AWS用户时所使用的账户就是就是我们根账户， 根账户就如同Linux上的root用户，能够对该用户在AWS上的所有资源进行完全、无限制的访问。我们一般不建议使用根账户去对AWS上的资源进行操作。

IAM能够做什么

试想一下，有这样一个组织，这个组织当中的所有人都使用同一个AWS账户下的资源，如果所有人都使用根用户对AWS上的资源进行操作，这样会面临什么样的问题呢？

这种做法带来的最直接也是最严重的问题就是安全方面的问题，数据被窃取，应用服务器被破坏，账户信息被泄露等等，这些安全问题可能来自于错误的操作，也可能来自于恶意的破坏。

让我们来看看IAM为我们提供了哪些功能：

  • 对你 AWS 账户的共享访问权限

    可以向其他人员授予管理和使用你 AWS 账户中的资源的权限，而不必共享你的密码或访问密钥。

  • 精细化权限

    可以针对不同资源向不同人员授予不同权限。同时，我们应当遵循授予最小权限这一标准的安全建议，即仅授予执行任务所需的最小权限。

  • 与很多的AWS服务集成

    比如，可以控制 Amazon EC2 上运行的应用程序针对 AWS 资源的安全访问权限。

值得一提的是，IAM是一项可以免费使用的服务。

你可以通过AWS 管理控制台（UI），命令行工具（aws cli）, 软件开发工具包 (sdk 编程式) 以及HTTPS API 使用IAM服务。

如何使用IAM

AWS提供了用户（users）、策略(policies)、群组（groups）和角色（roles）供AWS的客户来管理自己账号下的资源。

用户（users）

在AWS里，user本质上与Linux系统中的user是非常相似的概念。我们可以创建一个新的user，为其分配username和password，授予适当的权限，并在不再使用的时候将其删除.

我们可以在账户中创建与组织中的用户对应的 IAM 用户，而不是与他人共享根账户凭证。

IAM 用户不是单独的账户；它们是你账户中的用户。

策略（policies）

现在,我们已经创建了一个名为Bob的IAM user,但默认情况下Bob这名用户无法访问你在AWS中的任何资源.

这时，我们就需要使用策略（policy）来对用户访问进行授权。简而言之，策略（policy）就是用来描述谁在什么条件下能对哪些资源进行哪些操作。在AWS中，策略（policies）被分为AWS管理的策略和客户自己管理的策略。我们可以使用JSON文档来描述一个策略，

群组（groups）

Bob已经能够以只读的方式访问my-bucket这个存储桶里的资源了，但现在Bob的同事们也需要这个权限，我们当然可以一个一个的为他们授予这个权限，但这是一种很低效同时也很不便于管理的方式。我们可以使用群组（groups）来对Bob和他的同事进行管理。在AWS中，group的概念和Linux系统中group的概念也是很相似的，当你给一个group授权时，整个group里的user都会自动获得该权限；你将一个新用户添加到某一个group里，该用户也会自动获得该group拥有的权限。

你可以根据你需要该组的用户执行的工作类型为组分配权限，然后将用户添加到该组。

角色（roles）

IAM 角色 类似于用户，因为它是一个 AWS 实体，该实体具有确定其在 AWS 中可执行和不可执行的操作的权限策略。但是，角色旨在让需要它的任何人代入，而不是唯一地与某个人员关联。此外，角色没有任何关联的凭证（密码或访问密钥）。
我们可以使用角色向通常无权访问你的 AWS 资源的用户、应用程序或服务提供访问权限。

IAM 最佳实践

下面这些都是AWS官方提出的一些最佳实践：

  • 隐藏您的 AWS 账户（根）访问密钥
  • 创建单独的 IAM 用户
  • 使用组向 IAM 用户分配权限
  • 授予最小权限
  • 为您的用户配置强密码策略
  • 为特权用户启用 MFA
  • 针对在 Amazon EC2 实例上运行的应用程序使用角色
  • 通过使用角色而非共享证书来委托访问
  • 定期交替轮换证书
  • 删除不需要的证书
  • 使用策略条件来增强安全性
  • 监控 AWS 账户中的活动
